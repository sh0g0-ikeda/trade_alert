# 2. ターゲットユーザー

このプロダクトは、以下の条件に該当するユーザーを中心に設計する。
すべての UI / 機能 / プラン制限は、このターゲット層の行動・リテラシーに最適化する。

---

## 2.1 ユーザー属性（定義）

### **A. ライト投資家（中心ターゲット）**

* 年齢：20〜40代
* 職業：会社員（非エンジニア職）・大学生
* 投資歴：1〜3年程度
* 投資スタイル：スイング〜中長期
  → 数時間〜数日チャートを見ないことがある
* 保有資産：

  * 米国株を中心に数銘柄
  * 仮想通貨を少額
  * FXや金は持たない or 持っても少額
* 課題：

  * “仕事・学校でスマホを見られない間に大きく動いている” に気づけない
  * 証券アプリを複数併用しており、**監視のための画面が分散している**
* アプリへの要求：

  * シンプルでいい、細かい設定やチャートは不要
  * **「大きく動いたら通知だけほしい」**

### **B. 中級投資家**

* 投資歴：3〜5年
* 複数の資産クラス（株・仮想通貨・FX）を運用
* 監視銘柄：10〜30銘柄
* 課題：

  * 個別アプリを開いて監視するのが面倒
  * 通知条件を細かく調整したい（+3%、−8%など）
* アプリへの要求：

  * 通知頻度と閾値を**細かく調整**したい
  * 無料プランでは不十分 → **有料プラン誘導の中心層**

---

## 2.2 ユーザーの行動特性（実装判断に直結する要件）

### **① 1日のうち多くの時間“チャートを見ていない”**

* 想定行動：

  * 仕事中はチャートを開かない
  * 授業中・移動中はアプリを見ない
* 実装への影響：

  * 通知こそが価値
  * UI複雑化ではなく「放置運用」に最適化
  * アプリを毎日開かない前提で、再ログイン不要設計（トークン長期化）

---

### **② 投資用アプリに“情報が散らばっている”**

* 資産が

  * 楽天証券
  * SBI証券
  * GMO
  * コインチェック
  * Binance
  * DMM FX
    等にバラけているケースが多い。
* 実装視点：

  * 証券会社と OAuth 連携は行わない
  * **ユーザーが自分で銘柄だけ登録する UI** を最優先
  * スクショ一括入力機能が刺さる理由もここ

---

### **③ 数字や投資用語に強くないため“設定はシンプルである必要がある”**

* 無料プランは固定 ±5% にすることで

  * 設定の理解を不要にする
  * 閾値入力のミスを防ぐ
* 有料プランでは

  * 1%刻み + プリセット
  * 「上昇」「下落」を明確に分ける UI が必要

---

### **④ 過剰通知は嫌う**

* 想定行動：

  * LINE・メールなどの通知も多く、投資系通知は埋もれやすい
  * 「通知多すぎる＝即アンインストール」の傾向が強い
* 実装への影響：

  * クールダウンの仕組みが必須（特に仮想通貨）
  * 過敏な設定をしないよう注意喚起
  * 無料版は通知頻度を抑えて「手堅い価値提供」をする

---

### **⑤ 本気のデイトレ勢はターゲット外**

* 1分足の動きで売買 → こういう層は TradingView や証券アプリを使う
* したがって：

  * サーバー負荷が増える “秒〜分間隔の更新” はやらない
  * チャート搭載もしない

---

## 2.3 ユースケース（仕様の根拠になる実例）

### **ケース1：社会人（メインターゲット）**

* 9:00〜18:00の勤務中はほぼチャートを見ない
* 仕事が忙しく、アプリは1〜2日開かないこともある
* その間に株価が−7%動いて損切りタイミングを逃す
  → **通知されていれば判断できた**

必要仕様

* バックグラウンドで自動監視
* プッシュ通知中心設計
* 通知タップ後の画面は「価格と変動率」だけで良い

---

### **ケース2：大学生の投資初心者**

* 米株を数銘柄＋BTCを少額持っている
* 初めて投資したため、損切り判断が上手くできない
* 設定が複雑だと使わない
  → **無料プランの ±5% 固定がそのまま最適**

必要仕様

* 設定項目を極限まで少なく
* 銘柄検索・追加を最短で終わらせる
* スクショ入力があると初期登録が楽

---

### **ケース3：中級投資家**

* 監視銘柄10〜20
* 仮想通貨はボラが高いため、+10%、−12% のように細かく条件を分けたい
* 株とFXで条件を変えたい

必要仕様

* 有料プランは閾値を上下個別設定
* 資産クラスごとのプリセット
* 過剰通知抑止のクールダウン設定

---

## 2.4 ターゲットから設計に落ちる仕様の指針

ターゲット層の特徴から、以下のように仕様を固定する。

| ユーザー特性       | 仕様・設計への落とし込み                  |
| ------------ | ----------------------------- |
| チャート見ない時間が長い | 通知が最重要 → 24時間稼働のバッチをバックエンドで実装 |
| 銘柄が分散している    | 銘柄マスタをもつ・ユニーク銘柄管理必須           |
| 設定が苦手        | 無料プランは固定条件 / 有料でもプリセット中心      |
| 過剰通知を嫌う      | クールダウン処理必須・同一レベルの変動重複通知禁止     |
| 投資アプリ複数持ち    | スクショ一括入力で登録作業軽減               |

---

## 2.5 このターゲット設定の意図（ビジネス観点）

* 広すぎる市場（デイトレ〜仮想通貨専業など）を狙うと機能要求が爆発する
  → コスト増・実装遅延
  → App Store 審査も重くなる

* 「通知特化」「放置運用」ユーザーに絞ることで
  → UI・API・バッチ頻度などが最適化される
  → コスト構造が明確化する
  → 有料プランの価値がわかりやすくなる
